## 一、基础目标（必选）

### 2.1.1 交易转发

#### 需求 1：实现模拟交易转发逻辑，接受输入交易订单，输出交易订单

**✅ 已实现**

**实现位置**：
- **trading-gateway**：`GatewayHandler`（`com.trading.gateway.server.GatewayHandler`）
- **trading-core**：`OrderController.placeOrder` → `OrderService.placeOrder`

**实现方式**：
1. **Gateway 接收**：TCP 端口 9000，按行接收 JSON；`GatewayHandler.channelRead0` 解析 JSON，若无 `origClOrderId` 则解析为 `OrderRequest`。
2. **转发到 Core**：通过 `CoreClient.placeOrder(req)`（HTTP POST `http://localhost:8081/api/order`）转发。
3. **Core 处理**：`OrderService.placeOrder` 做校验与风控，通过后调用 exchange-core 引擎，将引擎回报转为 `OrderAckResponse` 返回。
4. **回报回写**：Gateway 将 Core 返回的 JSON 原样写回客户端（一行）。

**代码路径**：
- Gateway：`trading-gateway/src/main/java/com/trading/gateway/server/GatewayHandler.java`（第 40-65 行）
- Core：`trading-core/src/main/java/com/trading/core/service/OrderService.java`（第 87-185 行）

#### 需求 2：实现模拟回报转发逻辑，接受输入回报，输出回报

**✅ 已实现**

**实现位置**：
- **trading-core**：`OrderService.placeOrder` / `OrderService.cancelOrder` 构造回报并返回
- **trading-gateway**：将 Core 返回的 JSON 写回客户端

**实现方式**：

**代码路径**：

#### 需求 3：实现基本的交易校验功能，对于简单的无效订单，输出一条交易非法的回报

**✅ 已实现**

**实现位置**：
- **trading-core**：`OrderValidation`（`com.trading.core.risk.OrderValidation`）

**实现方式**：
- `OrderValidation.validate(OrderRequest)` 校验：
  - market ∈ {XSHG, XSHE, BJSE}
  - securityId 非空且 ≤ 6 字符
  - side 为 B 或 S
  - qty > 0
  - price > 0
  - shareholderId 非空且 ≤ 10 字符
  - clOrderId 若提供则 ≤ 64 字符（可不提供，系统自动生成）
- 不通过则返回 `OrderRejectResponse`（含 rejectCode、rejectText），不进入引擎。

**代码路径**：
- `trading-core/src/main/java/com/trading/core/risk/OrderValidation.java`（第 18-43 行）

### 2.1.3 模拟撮合

#### 需求 1：实现对敲交易订单端自动撮合，返回成交回报

**✅ 已实现**

**实现位置**：
- **trading-core**（依赖 **exchange-core**）：`OrderService.placeOrder` → `ExchangeEngineHolder.getApi().submitCommandAsyncFullResponse(ApiPlaceOrder)`
- **exchange-core**：`OrderBookNaiveImpl` / `OrderBookDirectImpl`（撮合引擎）

**实现方式**：
1. **下单到引擎**：`OrderService` 构造 `ApiPlaceOrder`（GTC、BID/ASK），调用引擎 API 同步等待结果。
2. **引擎撮合**：exchange-core 按**价格优先、时间优先**撮合（见 `OrderBookNaiveImpl`：价格档 TreeMap 顺序 + 同档 LinkedHashMap FIFO）。
3. **成交回报**：引擎返回 `OrderCommand`（含 `matcherEvent` 链，类型为 TRADE），Core 将结果转为 `OrderAckResponse`。
4. **成交事件推送**：`TradeEventProcessor` 消费引擎 `resultQueue`，解析 TRADE 事件，向 Admin 推送 FILLED/PARTIALLY_FILLED 事件。

**代码路径**：
- Core 下单：`trading-core/src/main/java/com/trading/core/service/OrderService.java`（第 121-135 行）
- 引擎撮合：`exchange-core/src/main/java/exchange/core2/core/orderbook/OrderBookNaiveImpl.java`（`tryMatchInstantly` 方法）
- 成交处理：`trading-core/src/main/java/com/trading/core/service/TradeEventProcessor.java`（第 110-200 行）

#### 需求 2：被匹配的对手方交易订单需要从交易所侧撤回

**实现位置**：

- Core 不修改引擎代码，仅调用其 API；引擎内部逻辑保证被匹配的订单自动从订单簿移除。

**代码路径**：

#### 需求 2：在撮合时考虑行情信息：撮合时需要保证买价、卖价和对手方价格保持一致

**✅ 已实现**

**说明**：在 `OrderService.placeOrder` 中，对敲风控之后、引擎下单之前增加行情校验（`MarketPriceValidator`）。支持两种模式：**EXACT**（买价=ask、卖价=bid）与 **INSIDE**（买价≤ask、卖价≥bid，或单边无挂单时跳过校验），默认 INSIDE。若超出范围返回 `OrderRejectResponse`（如 rejectCode 1010）。

**实现位置**：`trading-core` → `MarketPriceValidator`、`OrderService.placeOrder`；配置 `trading.core.market-validation.mode`、`required`。

### 2.2.2 撤单支持

#### 需求 1：处理输入的撤单请求

**✅ 已实现**

**实现位置**：
- **trading-gateway**：`GatewayHandler`（含 `origClOrderId` 则解析为 `CancelRequest`）
- **trading-core**：`OrderController.cancelOrder` → `OrderService.cancelOrder`

**实现方式**：
1. **Gateway 接收**：TCP 9000，JSON 含 `origClOrderId` 则解析为 `CancelRequest`，调用 `CoreClient.cancelOrder(req)`。
2. **Core 校验**：`CancelValidation.validate` 校验 `origClOrderId` 非空。
3. **查找订单**：根据 `origClOrderId` 查 `clOrderIdToOrderId` 得引擎 `orderId`；若内存中没有，从 `OpenOrderStore` 获取 `engineOrderId`。
4. **调用引擎**：构造 `ApiCancelOrder`，调用 `api.submitCommandAsyncFullResponse(cmd)` 同步等待。
5. **结果处理**：成功则从 `OpenOrderStore` 移除、取消 TradeEventProcessor 注册、更新 SelfTradeChecker、向 Admin 发送 CANCELLED 事件、返回 `CancelAckResponse`（含 cumQty、canceledQty）。

**代码路径**：
- Gateway：`trading-gateway/src/main/java/com/trading/gateway/server/GatewayHandler.java`（第 48-51 行）
- Core：`trading-core/src/main/java/com/trading/core/service/OrderService.java`（第 187-260 行）

- **成交事件**：通过 `TradeEventProcessor` 向 Admin 推送 FILLED/PARTIALLY_FILLED 事件，Admin 落库到 `t_order_history`。

**若需独立成交回报接口**，可新增：
- `GET /api/orders/{clOrderId}/trades`：查询某订单的成交明细（execId、execQty、execPrice）。

**当前实现**：成交信息在 Admin 的 `t_order_history` 表中（filledQty、status），无独立的“成交回报”JSON 结构。

## 五、未实现功能的建议实现路径

### 1. 行情接入

**已实现**。当前实现：盘口由 OpenOrderStore 计算并写入 MarketDataStore；查询 `GET /api/market-data`；下单前由 MarketPriceValidator 校验（EXACT/INSIDE）。详见 `docs/行情接入实现方案.md`、`docs/core模块功能与接口说明.md`。

### 2. 成交回报独立接口

1. 新增 `Trade` 实体（execId、clOrderId、execQty、execPrice、execTime）。
2. 在 `TradeEventProcessor` 中，每笔成交时写入 Trade 表。
3. 提供 `GET /api/orders/{clOrderId}/trades` 查询成交明细。

### 3. 历史记录分析

1. 新增 `OrderAnalysisController`，提供统计接口。
2. 使用 SQL 聚合查询（SUM、COUNT、GROUP BY）或 JPA Criteria API。
3. 返回 JSON：`{"totalVolume": 1000, "totalAmount": 10500.0, "orderCount": 10, ...}`。
