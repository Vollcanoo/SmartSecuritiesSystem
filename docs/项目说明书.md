# 模拟股票交易对敲撮合系统 — 项目说明书

本文档面向新手，尽量用通俗语言说明本项目的用途、结构、如何运行以及如何与系统交互。读完后你可以：了解项目是做什么的、由哪些部分组成、如何在本地跑起来、如何用接口/命令行做下单和撤单。

---

## 一、项目是做什么的？

### 1.1 用一句话说

本项目是一个**模拟 A 股股票交易**的系统：可以接收“买入/卖出”订单，做基本校验和对敲风控后，把合法订单送进撮合引擎，并返回“订单确认”“订单非法”“撤单确认”等回报，方便理解交易转发、风控和撮合流程。

### 1.2 和真实交易的区别

- **模拟**：不连接真实交易所，不涉及真实资金与证券。
- **对敲**：同一股东号下，不能对同一只股票既挂买单又挂卖单（避免自成交），系统会拒绝这类订单并回报“对敲”非法。
- **撮合**：使用内置的 exchange-core 引擎做价格优先、时间优先的撮合，并生成成交回报。

### 1.3 项目目标（与需求文档对应）

- **交易转发**：接收交易订单并转发到撮合引擎；接收引擎回报并转成标准格式输出。
- **基本校验**：无效订单（如错误市场、错误买卖方向、数量/价格为 0）直接拒绝，并返回“交易非法”回报。
- **对敲风控**：会触发对敲的订单不进入撮合，并返回“对敲”非法回报（rejectCode=1002）。
- **模拟撮合**：合法订单进入引擎撮合，支持撤单；回报格式符合需求文档中的数据结构。

## 二、你需要提前准备什么？

### 2.1 软件环境

| 软件 | 用途 | 建议版本 |
|------|------|----------|
| JDK | 运行 Java 程序 | **全项目统一 JDK 17**（根 pom 已配置；exchange-core 依赖要求） |
| Maven | 编译、打包、运行 | 3.6+ |
| MySQL | Admin 模块使用，做用户管理与订单历史表；一键启动前需先启动并创建库 | 5.7 或 8.x |

**说明**：当前 Admin/Core 的 REST 接口**无登录与鉴权**，仅适合内网或本地测试；生产环境建议增加 API Key、JWT 或网关鉴权。MySQL 密码可通过环境变量 `DB_PASSWORD` 配置。

### 2.2 如何检查是否安装成功

在终端执行：

```bash
java -version    # 应看到 java version "17.x"
mvn -version    # 应看到 Maven 版本号
```

### 2.3 外部依赖：交易引擎 exchange-core

本项目的**交易核心**依赖一个外部引擎 `exchange-core`（groupId: `exchange.core2`），需要先安装到本机 Maven 仓库后才能完整编译运行 **trading-core**。

- 若你已有该引擎的源码或 jar，在引擎项目目录执行：`mvn install`。
- 确保引擎的 groupId 为 `exchange.core2`，artifactId 为 `exchange-core`，版本与项目根目录 `pom.xml` 或 `trading-core/pom.xml` 中一致（如 `0.5.4-SNAPSHOT`）。

若暂时没有安装 exchange-core，可以只运行 **trading-admin** 和 **trading-gateway**（网关会连不上 Core，但本地能验证网关启动与配置）。

## 三、项目由哪些部分组成？

项目是一个 **Maven 多模块** 工程，根目录下每个子文件夹代表一个“模块”，各自负责一块功能。

### 3.1 目录结构一览

```
Trading-system/
├── pom.xml                 # 根 POM：定义所有子模块和公共依赖版本
├── start-all.sh            # 一键启动脚本（Mac/Linux）
├── stop-all.sh             # 一键停止脚本（Mac/Linux）
├── start-all.bat           # 一键启动脚本（Windows）
├── stop-all.bat            # 一键停止脚本（Windows）
├── docs/                   # 文档目录
│   ├── 项目说明书.md       # 本文档
├── trading-common/        # 公共模块：订单/回报的数据结构、枚举等
├── trading-protocol/      # 协议相关：如网关消息类型枚举
├── trading-admin/         # 管理后台：用户（股东）管理，REST API
├── trading-core/          # 交易核心：校验、风控、调用撮合引擎、REST API
└── trading-gateway/       # 交易网关：接收客户端 TCP 连接，转发到 Core
```

### 3.2 各模块是干什么的？（新手向）

| 模块 | 通俗解释 | 主要技术 |
|------|-----------|----------|
| **trading-common** | 定义“订单长什么样”“回报长什么样”（字段名、类型），供其他模块共用 | Java Bean、枚举 |
| **trading-protocol** | 定义网关和客户端之间的消息类型（如 ORDER、CANCEL），便于扩展 | 枚举 |
| **trading-admin** | 后台管理：用户（股东）管理、订单历史表维护与查询；接收 Core 推送的订单事件落库 | Spring Boot、JPA、MySQL |
| **trading-core** | 交易核心：接收下单/撤单请求 → 做基本校验和对敲风控 → 调用 exchange-core 撮合 → 把引擎结果转成标准回报 | Spring Boot、exchange-core |
| **trading-gateway** | 交易网关：客户端通过 TCP 连上来，发一行 JSON；网关识别是下单还是撤单，转发给 Core，再把 Core 的回报原样发回客户端 | Spring Boot、Netty |

### 3.3 数据流（谁连谁）

```
[ 客户端程序 ]  --TCP 9000, 一行 JSON-->  [ trading-gateway ]  --HTTP 8081-->  [ trading-core ]
                                                                                        |
                                                                                        v
                                                                               [ exchange-core 引擎 ]

[ 管理端/脚本 ]  --HTTP 8080-->  [ trading-admin ]  --数据库-->  [ MySQL ]

[ trading-core ]  --订单事件 POST /internal/order-events-->  [ trading-admin ]  -->  t_order_history 表
```

- **交易链路**：客户端 → 网关(9000) → Core(8081) → 引擎。  
- **管理链路**：浏览器或 curl 访问 Admin(8080)，管理用户、查询订单历史（历史订单表在 Admin，由 Core 推送事件落库）。
- **订单历史**：Core 在下单/撤单/成交时向 Admin 推送事件，Admin 写入 `t_order_history`；查询历史走 Admin 的 `GET /api/orders/history`；支持 `DELETE /api/orders/history/{id}` 删单条、`DELETE /api/orders/history` 删全部。

## 四、如何编译与运行？

### 4.1 一键启动（推荐）

在项目根目录执行一条命令即可**按顺序编译并启动** Admin、Core、Gateway 三个服务，无需分步启动。

**Mac / Linux：**

```bash
cd /path/to/Trading-system
./start-all.sh
```

- 脚本会先执行 `mvn package -DskipTests` 打包，再依次启动 trading-admin → trading-core → trading-gateway，并等待各端口就绪后再启动下一项。
- **trading-admin 使用 MySQL**：一键启动前请先**启动 MySQL 并创建数据库** `trading_admin`，并在 `trading-admin/src/main/resources/application.yaml` 中配置好 `url`、`username`、`password`。
- 启动完成后会提示各服务地址；进程 PID 写入 `.trading-system.pids`。
- 日志输出到根目录：`logs-admin.log`、`logs-core.log`、`logs-gateway.log`。
- **停止全部服务**：在同一目录执行 `./stop-all.sh`。

**Windows：**

```cmd
cd /d D:\path\to\Trading-system
start-all.bat
```

- 会先打包，再在**三个新窗口**中分别启动 Admin、Core、Gateway（便于查看各自日志）。
- 关闭对应窗口即可停止该服务；或运行 `stop-all.bat` 尝试按窗口标题结束进程。

**注意：**

- 一键启动前请确保已安装 **exchange-core** 并执行过 `mvn install`，否则 trading-core 打包会失败，脚本会报错退出。
- **全项目使用 JDK 17**：请在本地安装 JDK 17 并设为默认 `java`（或设置 `JAVA_HOME`），编译与运行均使用 17。
- **MySQL 需先就绪**：Admin 依赖 MySQL，请先启动 MySQL、创建数据库 `trading_admin`、并配置好 application.yaml 中的连接信息。
- 若只需运行 Admin（不跑 Core/Gateway），可仍按下面 4.3 节分步启动。

**多机部署与健康检查：**

- **多机部署**：Core 的 `trading.admin.base-url`、Gateway 的 `trading.gateway.core-base-url` 默认为本机；若 Admin/Core 部署在不同机器，请通过配置文件或环境变量（如 `ADMIN_BASE_URL`）修改。
- **健康检查**：各服务均暴露 Spring Boot Actuator 的 `health` 端点，便于运维或网关探测：  
  Admin `http://localhost:8080/actuator/health`，Core `http://localhost:8081/actuator/health`，Gateway `http://localhost:8082/actuator/health`（Gateway 的 HTTP 端口为 8082，TCP 业务端口为 9000）。

### 4.2 编译整个项目（仅编译不运行）

在项目根目录执行：

```bash
cd /path/to/Trading-system
mvn clean compile
```

- 若未安装 exchange-core，编译 **trading-core** 会报错，可暂时只编译其他模块：  
  `mvn clean compile -pl trading-common,trading-protocol,trading-admin,trading-gateway -am`

### 4.3 分步运行各服务（未使用一键启动时）

1. **MySQL**（Admin 使用，必选）  
   - 创建数据库：`CREATE DATABASE trading_admin;`  
   - 在 `trading-admin/src/main/resources/application.yaml` 中配置 `url`、`username`、`password`。

2. **trading-admin（管理后台）**  
   ```bash
   cd trading-admin
   mvn spring-boot:run
   ```  
   - 默认端口：**8080**。  
   - 若提示“端口被占用”，可改为其他端口，例如：  
     `mvn spring-boot:run -Dspring-boot.run.arguments=--server.port=8083`

3. **trading-core（交易核心）**  
   ```bash
   cd trading-core
   mvn spring-boot:run
   ```  
   - 默认端口：**8081**。  
   - 依赖 exchange-core，需先安装到本地 Maven。

4. **trading-gateway（交易网关）**  
   ```bash
   cd trading-gateway
   mvn spring-boot:run
   ```  
   - 客户端连接端口：**9000**（在 `application.yaml` 中 `trading.gateway.port`）。  
   - 网关会请求 Core 的地址由 `trading.gateway.core-base-url` 指定，默认 `http://localhost:8081`。  
   - 先启动 Core，再启动 Gateway。

### 4.4 端口汇总（默认）

| 服务 | 端口 | 说明 |
|------|------|------|
| trading-admin | 8080 | HTTP，用户管理 API |
| trading-core | 8081 | HTTP，下单/撤单 API |
| trading-gateway | 8082 | Spring 内嵌 Web（可选） |
| trading-gateway | 9000 | TCP，客户端连接，按行 JSON |

## 五、接口与数据格式说明

以下所有 JSON 字段名与需求文档一致，便于对照。

### 5.1 约定：市场与买卖方向

- **市场 market**：只接受以下三选一（字符串）。  
  - `XSHG`：上交所  
  - `XSHE`：深交所  
  - `BJSE`：北交所  

- **买卖方向 side**：  
  - `B`：买  
  - `S`：卖  

其他 market 或 side 会在 Core 被校验为非法，并返回“订单非法”回报。

### 5.2 交易核心（trading-core）REST API

Base URL：`http://localhost:8081`（默认）。

#### 5.2.1 下单

- **地址**：`POST /api/order`  
- **请求体**：交易订单（JSON），见下表。  
- **返回**：HTTP 200，body 为 **订单确认回报** 或 **订单非法回报**（JSON）。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| clOrderId | string | 是 | 订单唯一编号，建议 ≤16 字符 |
| market | string | 是 | XSHG / XSHE / BJSE |
| securityId | string | 是 | 股票代码，如 "600030"，≤6 字符 |
| side | string | 是 | B 或 S |
| qty | number | 是 | 数量，必须 > 0 |
| price | number | 是 | 价格，必须 > 0 |
| shareholderId | string | 是 | 股东号，≤10 字符 |

**示例（curl）：**

```bash
curl -X POST http://localhost:8081/api/order \
  -H "Content-Type: application/json" \
  -d '{
    "clOrderId": "ord001",
    "market": "XSHG",
    "securityId": "600030",
    "side": "B",
    "qty": 100,
    "price": 10.5,
    "shareholderId": "A123456789"
  }'
```

**成功时**返回订单确认回报，包含 `clOrderId`、`market`、`securityId`、`side`、`qty`、`price`、`shareholderId`、`orderId`（系统内部订单号）等。  
**失败时**返回订单非法回报，包含 `rejectCode`、`rejectText`，例如：

- `rejectCode: 1006`、`rejectText: "市场代码须为 XSHG/XSHE/BJSE"`  
- `rejectCode: 1002`、`rejectText: "对敲交易，同股东号下存在反向未成交订单"`  

#### 5.2.2 撤单

- **地址**：`POST /api/cancel`  
- **请求体**：撤单请求（JSON），见下表。  
- **返回**：HTTP 200，body 为 **撤单确认回报** 或 **撤单非法回报**（JSON）。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| clOrderId | string | 是 | 本笔撤单的唯一编号 |
| origClOrderId | string | 是 | 要撤掉的那笔订单的 clOrderId（下单时填的那个） |
| market | string | 是 | 原订单市场 |
| securityId | string | 是 | 原订单股票代码 |
| shareholderId | string | 是 | 原订单股东号 |
| side | string | 是 | 原订单方向 B/S |

**示例（curl）：**

```bash
curl -X POST http://localhost:8081/api/cancel \
  -H "Content-Type: application/json" \
  -d '{
    "clOrderId": "cancel001",
    "origClOrderId": "ord001",
    "market": "XSHG",
    "securityId": "600030",
    "shareholderId": "A123456789",
    "side": "B"
  }'
```

### 5.3 管理后台（trading-admin）REST API

Base URL：`http://localhost:8080`（默认）。  
所有成功响应通常形如：`{ "code": 0, "message": "success", "data": ... }`。

| 功能 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 创建用户 | POST | /api/users | Body: 仅 **username**，系统生成 uid、shareholderId |
| 修改用户名 | PUT | /api/users/{uid}/username | Body: { "username": "新用户名" } |
| 查询用户余额 | GET | /api/users/{uid}/balance | 返回资金与持仓（简化实现） |
| 查询所有用户 | GET | /api/users | 返回用户列表 |
| 按 uid 查询 | GET | /api/users/{uid} | 路径为数字 uid |
| 按股东号查询 | GET | /api/users/shareholder/{shareholderId} | 路径为股东号 |
| 禁用用户 | POST | /api/users/{uid}/suspend | 路径为 uid |
| 启用用户 | POST | /api/users/{uid}/resume | 路径为 uid |
| 订单历史查询 | GET | /api/orders/history | 可选 shareholderId、status、page、size |
| 删除单条历史订单 | DELETE | /api/orders/history/{id} | 按主键 id 删除，成功 204 |
| 删除全部历史订单 | DELETE | /api/orders/history | 清空历史，返回 { deletedCount } |

**创建用户示例：**

```bash
curl -X POST http://localhost:8080/api/users \
  -H "Content-Type: application/json" \
  -d '{"username":"张三"}'
```

### 5.4 交易网关（trading-gateway）TCP 协议

- **连接**：TCP 连接到本机 **9000** 端口。  
- **协议**：**一行一条 JSON**，以换行符结束。  
- **区分下单/撤单**：  
  - 若 JSON 里包含字段 **origClOrderId**，视为**撤单**；  
  - 否则视为**下单**。  
- **请求**：下单用“交易订单”同一结构（与 5.2.1 一致）；撤单用“撤单请求”同一结构（与 5.2.2 一致）。  
- **响应**：网关把 Core 返回的 JSON **原样**再发回客户端，也是**一行**一条 JSON。

**示例（用 nc 或 telnet）：**

```bash
# 下单（发送一行，收到一行回报）
echo '{"clOrderId":"g1","market":"XSHG","securityId":"600030","side":"B","qty":100,"price":10.5,"shareholderId":"A123"}' | nc localhost 9000

# 撤单
echo '{"clOrderId":"c1","origClOrderId":"g1","market":"XSHG","securityId":"600030","shareholderId":"A123","side":"B"}' | nc localhost 9000
```

## 六、回报类型与错误码（需求文档对应）

### 6.1 回报种类

| 回报类型 | 说明 | 典型字段 |
|----------|------|----------|
| 订单确认 | 交易所已接受订单 | clOrderId, orderId, market, securityId, side, qty, price, shareholderId |
| 订单非法 | 订单被拒绝（校验或对敲） | 同上 + rejectCode, rejectText |
| 撤单确认 | 撤单成功 | clOrderId, origClOrderId, market, securityId, cumQty, canceledQty 等 |
| 撤单非法 | 撤单被拒绝（如订单不存在） | clOrderId, origClOrderId, rejectCode, rejectText |

### 6.2 非法回报错误码（rejectCode）

| 代码 | 含义 |
|------|------|
| 1001 | 无效订单（如格式、必填项） |
| 1002 | 对敲交易（同股东号下存在反向未成交订单） |
| 1003 | 资金不足（当前引擎未做资金检查时可作预留） |
| 1004 | 订单不存在（常见于撤单时 origClOrderId 有误或未下单成功） |
| 1005 | 无效证券 |
| 1006 | 无效市场（非 XSHG/XSHE/BJSE） |
| 1007 | 无效价格（如 ≤0） |
| 1008 | 无效数量（如 ≤0） |
| 1009 | 无效股东号 |

## 八、如何自测“符合需求文档”？

### 8.1 基本校验

- 对 Core 发一笔**非法订单**，例如：  
  - `market` 填 `"SH"`（非法市场）→ 应返回订单非法回报，`rejectCode` 为 1006。  
  - `qty` 填 `0` → 应返回订单非法回报，`rejectCode` 为 1008。

### 8.2 对敲风控

1. 用**同一股东号**、**同一市场+证券**（如 XSHG + 600030），先下一笔**买**单，确认返回订单确认。  
2. 不撤单、不再成交的前提下，再下一笔**卖**单（同一 shareholderId、market、securityId）。  
3. 第二笔应被拒绝，返回订单非法回报，`rejectCode=1002`（对敲），且该笔不会进入撮合。

### 8.3 撤单

1. 先下一笔单，记下返回的 `clOrderId`。  
2. 用该 `clOrderId` 作为 `origClOrderId` 发撤单请求。  
3. 应返回撤单确认回报，且 `canceledQty` 等字段合理。

## 九、常见问题（FAQ）

**Q1：编译时报错找不到 exchange.core2:exchange-core**  
- 说明本地 Maven 仓库没有该引擎。需要先拿到 exchange-core 的源码或 jar，在对应项目中执行 `mvn install`，再编译本工程。  

**Q2：启动 trading-admin 报错 “Port 8080 is already in use”**  
- 本机 8080 已被占用。可修改 `application.yaml` 的 `server.port`，或用：  
  `mvn spring-boot:run -Dspring-boot.run.arguments=--server.port=8083` 指定端口。  

**Q3：Gateway 连不上 Core**  
- 确认 Core 已先启动；确认 `trading.gateway.core-base-url` 与 Core 实际地址、端口一致（如 `http://localhost:8081`）。  

**Q4：撤单报“订单不存在”**  
- `origClOrderId` 必须与下单成功时使用的 `clOrderId` 完全一致，且该笔订单确实已被 Core 接受（曾返回订单确认）。  

**Q5：market 填 SH/SZ 可以吗？**  
- 不可以。需求文档规定仅接受 XSHG（上交所）、XSHE（深交所）、BJSE（北交所），填其他值会返回 1006 非法市场。

## 十、术语表（方便新手查阅）

| 术语 | 解释 |
|------|------|
| 对敲 | 同一股东号下，对同一只股票既挂买单又挂卖单，可能形成自己和自己成交；本系统禁止此类订单。 |
| 撮合 | 按价格优先、时间优先等规则，将买单和卖单匹配成交的过程。 |
| 股东号 | 交易所或券商分配给投资者的唯一标识，本系统中即 shareholderId。 |
| clOrderId | 客户（或上游系统）为每笔订单起的唯一编号，最长 16 字符。 |
| orderId | 系统内部（或引擎）为订单生成的唯一编号，用于撤单时在引擎侧定位订单。 |
| 回报 | 系统对订单/撤单处理结果的反馈，如订单确认、订单非法、撤单确认、撤单非法等。 |
| 交易转发 | 接收客户订单并转交给撮合引擎，再把引擎结果转成标准回报返回。 |
