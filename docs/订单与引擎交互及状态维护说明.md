# 订单与引擎交互及状态维护说明

本文档回答：**下单/撤单是否调用 exchange-core 引擎**、**订单撮合之后本项目如何维护订单状态**，以及**撮合是否体现「价格优先、时间优先」**。

---

## 一、下单是否调用 exchange-core？

**是。** 下单流程会调用 exchange-core 的**下单**接口。

- **入口**：`OrderController.placeOrder` → `OrderService.placeOrder`
- **调用引擎**：构造 `ApiPlaceOrder`（orderId、price、size、action、symbol、uid 等），通过 `ExchangeEngineHolder.getApi().submitCommandAsyncFullResponse(cmd)` **同步等待**引擎结果。
- **引擎作用**：将订单写入订单簿、若可撮合则立即撮合，返回 `OrderCommand`（含 resultCode、matcherEvent 等）。引擎内部维护订单簿与撮合逻辑。

---

## 二、撤单是否调用 exchange-core？

**是。** 撤单流程会调用 exchange-core 的**撤单**接口。

- **入口**：`OrderController.cancelOrder` → `OrderService.cancelOrder`
- **调用引擎**：根据业务侧 `origClOrderId` 解析出引擎的 `orderId`，构造 `ApiCancelOrder`（orderId、uid、symbol），通过 `api.submitCommandAsyncFullResponse(cmd)` 提交给引擎并同步等待。
- **引擎作用**：在订单簿中查找该 orderId，执行撤单（引擎内为 REDUCE/CANCEL），从订单簿移除或减少数量，并产生相应的 `OrderCommand`（如 matcherEvent 类型为 REDUCE）。引擎返回后，Core 根据 resultCode 判断成功与否。
- **代码位置**：`trading-core` 中 `OrderService.cancelOrder` 内：
  ```java
  ApiCancelOrder cmd = ApiCancelOrder.builder()
      .orderId(engineOrderId)
      .uid(uid)
      .symbol(symbolId)
      .build();
  CompletableFuture<OrderCommand> future = api.submitCommandAsyncFullResponse(cmd);
  OrderCommand result = future.get(TIMEOUT_MS, TimeUnit.MILLISECONDS);
  ```
- **成功后 Core 侧处理**：从 `OpenOrderStore` 移除该挂单、从 `clOrderIdToOrderId` 移除映射、在 `TradeEventProcessor` 中取消 orderId→clOrderId 注册、更新 `SelfTradeChecker` 未成交量、向 Admin 发送 CANCELLED 事件。

因此，**撤单与下单一样，都会真正调用 exchange-core 引擎**，由引擎从订单簿撤单；本项目不在未调引擎的情况下“本地撤销”订单。

---

## 三、订单撮合之后如何维护订单状态？

本项目采用**引擎 + 业务双轨**：引擎负责订单簿与撮合，业务（trading-core + trading-admin）负责挂单视图与历史状态。

### 3.1 引擎侧（exchange-core）

- 维护**订单簿**（买/卖盘、价格档位、订单队列）。
- **撮合时**：产生 **TRADE** 类型事件（matcherEvent），挂在对应 `OrderCommand` 上；同一笔 Command 可能含多笔成交（taker 与多个 maker）。
- **撤单时**：产生 **REDUCE** 类型事件。
- 每笔命令执行完成后，通过 `ExchangeEngineHolder` 注册的 `resultsConsumer` 将 `OrderCommand`（含 matcherEvent）放入 **resultQueue**，供业务侧消费。

### 3.2 业务侧（trading-core）

- **挂单视图（OpenOrderStore）**
  - **写入**：下单成功且未完全成交时，将订单写入 `OpenOrderStore`（按 clOrderId 存储，含 shareholderId、market、securityId、side、price、orderQty、filledQty、engineOrderId）。
  - **移除**：
    - **撤单成功**：在 `OrderService.cancelOrder` 中直接 `openOrderStore.remove(origClOrderId)`。
    - **完全成交**：由 **TradeEventProcessor** 消费 `resultQueue` 中的 `OrderCommand`，解析 `matcherEvent` 中的 **TRADE** 事件，对 taker 与 maker 分别更新/移除：
      - 根据 `orderId` / `matchedOrderId` 找到业务 `clOrderId`（通过 orderIdToClOrderId 或 OpenOrderStore.findClOrderIdByEngineOrderId）；
      - 更新该订单在 OpenOrderStore 中的 `filledQty`，若已完全成交则从 OpenOrderStore **移除**，并从映射中注销；
      - 同时向 Admin 发送 **FILLED** 或 **PARTIALLY_FILLED** 事件（含 clOrderId、filledQty 等）。
  - **查询**：`GET /api/orders?shareholderId=xxx` 来自 `OpenOrderStore.listByShareholder`，因此只显示当前未撤单、未完全成交的订单；**撮合完成后，已完全成交的订单会从挂单列表消失**，正是由上述移除逻辑保证。

- **对敲未成交量（SelfTradeChecker）**
  - 下单成功时按方向增加未成交量，撤单成功时扣减；用于禁止同股东号同市场同证券下反向挂单（对敲风控）。

- **订单 ID 映射**
  - `clOrderIdToOrderId`：业务 clOrderId → 引擎 orderId，供撤单时查找 engineOrderId。
  - `TradeEventProcessor.orderIdToClOrderId`：引擎 orderId → 业务 clOrderId，供 resultQueue 回报中根据 engineOrderId 找到业务订单并更新挂单/发送事件。

### 3.3 历史订单状态（trading-admin）

- **数据来源**：Core 在以下时机向 Admin 推送订单事件（`POST /internal/order-events`）：
  - 下单成功：**PLACED**（或若当即完全成交则为 **FILLED**）；
  - 撤单成功：**CANCELLED**；
  - 成交（由 TradeEventProcessor 发送）：**PARTIALLY_FILLED** / **FILLED**。
- **落库与更新**：Admin 的 `OrderHistoryService.processEvent` 根据事件类型与 `clOrderId` 插入或更新 `t_order_history` 表（filledQty、status=LIVE/PARTIALLY_FILLED/FILLED/CANCELLED 等），从而维护**历史订单状态**，供前端或接口按股东号/状态查询。

### 3.4 小结（撮合后状态维护）

| 维护项           | 位置           | 撮合后的更新方式 |
|------------------|----------------|------------------|
| 订单簿与成交结果 | exchange-core  | 引擎内部撮合、产生 TRADE 事件 |
| 挂单列表         | Core OpenOrderStore | TradeEventProcessor 消费 resultQueue，根据 TRADE 更新 filledQty，完全成交则移除 |
| 对敲未成交量     | Core SelfTradeChecker | 下单/撤单时已维护；成交量在 onOrderAccepted 的 filled 中体现 |
| 历史订单状态     | Admin t_order_history | Core 推送 PLACED/FILLED/PARTIALLY_FILLED/CANCELLED，Admin 落库更新 status、filledQty |

因此，**订单撮合之后**：引擎负责订单簿与成交结果；本项目通过 **TradeEventProcessor 消费引擎 resultQueue** 维护 Core 侧挂单视图（更新/移除），并通过 **OrderEventSender 向 Admin 推送事件** 维护历史订单状态。

## 四、撮合之后订单状态维护（详细流程）

### 4.1 引擎回报如何到达业务

1. 引擎每执行完一笔命令（下单/撤单等），会把 **OrderCommand**（含 `resultCode`、`matcherEvent`、可选 `marketData`）交给注册的 **resultsConsumer**。
2. 本项目中，`ExchangeEngineHolder.onResult` 把 `OrderCommand.copy()` **offer 到 resultQueue**（有界队列，满则丢弃并打日志）。
3. **TradeEventProcessor** 的后台线程在 `processEvents()` 里 **resultQueue.take()** 阻塞取命令，然后调用 `processOrderCommand(cmd)`。

### 4.2 单笔 OrderCommand 的处理逻辑（processOrderCommand）

- **无 matcherEvent**：直接 return（例如纯撤单成功的回报可能没有 TRADE，业务已在 cancelOrder 里处理过挂单移除）。
- **有 matcherEvent**：引擎把一次撮合产生的多笔成交串成一条 **MatcherTradeEvent 链**（nextEvent 指针），每条事件的 `eventType` 可能是 TRADE、REDUCE、REJECT 等；这里只关心 **TRADE**。
- **当前命令对应的订单**：`cmd.orderId` 是引擎订单 ID，通过 **orderIdToClOrderId.get(cmd.orderId)** 得到业务 **clOrderId**（即本笔命令的 taker 订单）。

对链上每一个 **TRADE** 事件：

1. **处理对手单（maker）**
   - `event.matchedOrderId` 是 maker 的引擎订单 ID。
   - 用 **orderIdToClOrderId.get(matchedOrderId)** 或 **openOrderStore.findClOrderIdByEngineOrderId(matchedOrderId)** 得到 maker 的 **clOrderId**。
   - 若找到，调用 **updateOrderFillStatus(makerClOrderId, matchedOrderId, event.size, event.matchedOrderCompleted)**，更新 maker 的挂单成交、完全成交则移除并发 FILLED/PARTIALLY_FILLED。

2. **累计当前订单（taker）的成交**
   - 若 **clOrderId != null**（当前订单还在映射里）：`totalFilled += event.size`，若 `event.activeOrderCompleted` 则标记 **orderCompleted = true**。
   - 若 taker 在 placeOrder 时已立即完全成交，可能已从映射中移除，此时 clOrderId 为 null，上面已处理 maker 即可。

3. **更新 taker 挂单并发送事件**
   - 若 **totalFilled > 0 且 clOrderId != null**，调用 **updateOrderFillStatus(clOrderId, cmd.orderId, totalFilled, orderCompleted)**，更新 taker 的 filledQty、完全成交则从 OpenOrderStore 移除并 unregister 映射，并发送 FILLED 或 PARTIALLY_FILLED 给 Admin。

### 4.3 updateOrderFillStatus（单笔订单的成交更新）

- 从 **OpenOrderStore.get(clOrderId)** 取挂单记录；若不存在（例如已在前置逻辑中移除），仍可调用 **sendFillEventWithoutRecord** 发事件，供 Admin 用已有历史记录更新。
- **currentFilled = record.getFilledQty()**，**newFilled = currentFilled + filledQty**，**record.setFilledQty(newFilled)**。
- 若 **orderCompleted** 或 **newFilled >= orderQty**：从 **OpenOrderStore.remove(clOrderId)**、**orderIdToClOrderId.remove(engineOrderId)**，并 **sendFillEvent(..., fullyFilled=true)**；否则 **sendFillEvent(..., fullyFilled=false)**。
- **sendFillEvent** 通过 **OrderEventSender.sendEvent** 推送到 Admin 的 **POST /internal/order-events**，事件体含 eventType、clOrderId、engineOrderId、shareholderId、market、securityId、side、price、orderQty、filledQty 等。

### 4.4 Admin 侧历史状态更新（processEvent）

- **OrderHistoryService.processEvent** 根据 **eventType** 分支：
  - **PLACED / PARTIALLY_FILLED / FILLED**：按 clOrderId 查库，若存在则更新 filledQty、根据 filled 与 orderQty 算 status（LIVE / PARTIALLY_FILLED / FILLED）并 save；若不存在则插入新记录。
  - **CANCELLED**：按 clOrderId 查库，若存在则设 status=CANCELLED、canceledQty 并 save。
- 因此，**撮合后的每一笔部分/完全成交**，都会通过 Core 的 FILLED/PARTIALLY_FILLED 事件驱动 Admin 更新 **t_order_history** 的 **filledQty** 和 **status**，实现历史订单状态的持久化维护。

## 五、引擎订单簿是否基于内存？

**是。** exchange-core 的订单簿是**纯内存**实现。

- **实现类**：如 **OrderBookNaiveImpl**、**OrderBookDirectImpl**（由 PerformanceConfiguration 的 orderBookFactory 决定）。
- **数据结构**（以 OrderBookNaiveImpl 为例）：
  - **askBuckets / bidBuckets**：`NavigableMap<Long, OrdersBucketNaive>`（TreeMap），按价格档位存放订单桶；
  - **idMap**：`LongObjectHashMap<Order>`，订单 ID → Order 对象。
- 订单、档位、队列均在 JVM 堆内，**无数据库或文件直接参与撮合路径**。引擎重启后订单簿为空，除非通过**快照恢复**或**日志回放**（见下节）重新构建。

## 六、外部如何获取订单簿信息并持久化？

### 6.1 引擎提供的“订单簿”接口

- **L2 快照**：引擎提供 **ApiOrderBookRequest(symbol, size)**。提交后，撮合引擎在执行该命令时调用 **orderBook.getL2MarketDataSnapshot(size)**，将结果填入 **OrderCommand.marketData**（类型为 **L2MarketData**），命令完成后同一 **OrderCommand** 经 **resultsConsumer** 回调给调用方。
- **L2MarketData** 内容：买卖档的 price、volume、orderCount 数组（askPrices、askVolumes、askOrders、bidPrices、bidVolumes、bidOrders），以及 askSize/bidSize、timestamp、referenceSeq 等，是**只读快照**，不包含逐笔订单明细。
- 若配置了“每笔命令附带 L2”（如某些风险模块），部分下单/撤单回报里也会带 **marketData**，但**请求 L2 的正式方式**仍是主动发 **ApiOrderBookRequest**。

### 6.2 引擎侧的持久化（可选）

- 引擎支持**可插拔的序列化**（**SerializationConfiguration**）：
  - **DEFAULT**：**DummySerializationProcessor**，不落盘、不写日志；
  - **DISK_SNAPSHOT_ONLY**：仅快照（订单簿/风险等状态序列化到磁盘）；
  - **DISK_JOURNALING**：快照 + 命令日志，可回放恢复订单簿与状态。
- **ISerializationProcessor** 负责：快照的 **storeData/loadData**、命令的 **writeToJournal**、**replayJournalFull** 等。订单簿通过 **IOrderBook.writeMarshallable** 序列化进快照。
- **本项目**：`ExchangeEngineHolder.buildConfig()` 使用 **ExchangeConfiguration.defaultBuilder()**，未替换 **serializationCfg**，因此是 **DEFAULT**，即 **DummySerializationProcessor**，**引擎内部不做任何订单簿或命令的持久化**。

### 6.3 本项目的“订单簿”与持久化方式

- **挂单列表**：来自 **trading-core** 的 **OpenOrderStore**（内存 ConcurrentHashMap），按 clOrderId 存 OpenOrderRecord；**不**从引擎拉取 L2 或逐笔订单列表。挂单的写入/移除完全由下单成功、撤单成功、以及 **TradeEventProcessor** 根据成交事件更新/移除。
- **历史订单**：来自 **trading-admin** 的 **t_order_history** 表（如 H2/MySQL），由 Core 推送的 **PLACED / FILLED / PARTIALLY_FILLED / CANCELLED** 事件驱动 **OrderHistoryService.processEvent** 做插入/更新。这是**业务侧**的订单状态持久化，**不是**引擎订单簿 L2 的持久化。
- **当前未使用 ApiOrderBookRequest**：trading-core 未调用引擎的 **ApiOrderBookRequest**，因此**没有**拉取 L2 快照并落库或展示的流程。若将来需要“盘口/深度”展示或盘后分析，可以：
  - 定时或按需调用 **api.submitCommandAsyncFullResponse(ApiOrderBookRequest.builder().symbol(symbolId).size(depth).build())**，从返回的 **OrderCommand.marketData** 取得 **L2MarketData**，再写入自有 DB 或文件；和/或
  - 为引擎配置 **SerializationConfiguration.DISK_SNAPSHOT_ONLY** 或 **DISK_JOURNALING**，在引擎侧做订单簿与命令的持久化与恢复（需使用相同的 ExchangeConfiguration 与 InitialStateConfiguration）。

## 七、撮合时的「价格优先、时间优先」是否体现？

需求文档要求模拟 A 股「价格优先、时间优先」的竞价撮合机制。**本项目在引擎层已实现该原则**，但业务文档与接口说明中未显式写出，容易让人误以为“没有体现”。下面说明实现位置，并给出可选的文档与测试改进。

### 7.1 引擎侧已实现的规则

本项目使用的订单簿为 **OrderBookNaiveImpl**（`ExchangeEngineHolder` 通过 `PerformanceConfiguration.baseBuilder()` 未改 `orderBookFactory`，默认即为 `OrderBookNaiveImpl::new`）。

- **价格优先（同一品种内）**
  - **买盘**：按**价格从高到低**排序（`bidBuckets = new TreeMap<>(Collections.reverseOrder())`），撮合时优先匹配**最高买价**。
  - **卖盘**：按**价格从低到高**排序（`askBuckets = new TreeMap<>()`），撮合时优先匹配**最低卖价**。
  - 限价单进入撮合时，通过 **subtreeForMatching(action, price)** 得到 `headMap(price, true)`，即“可成交的对手盘档位”；**tryMatchInstantly** 按 **matchingBuckets.values()** 顺序遍历（TreeMap 的 value 迭代顺序=键的升序/降序），因此**先扫最优价档、再扫次优价档**，实现价格优先。
- **时间优先（同一价格档内）**
  - 每个价格档对应一个 **OrdersBucketNaive**，内部使用 `LinkedHashMap<Long, Order>`（按插入顺序）存订单。
  - **put(order)** 时新订单插入 Map，**LinkedHashMap 保持插入顺序**。
  - **match(volumeToCollect, ...)** 中通过 **entries.entrySet().iterator()** 遍历，顺序即为插入顺序，即**先挂单先成交**（FIFO），实现同价下的时间优先。代码注释也写明：“starting from eldest records”。

因此：**价格优先**由“按档位最优价先撮合”体现，**时间优先**由“同档内按挂单先后顺序成交”体现；二者均在 **exchange-core** 的 **OrderBookNaiveImpl** 与 **OrdersBucketNaive** 中实现，本项目在调用引擎时**未改变**该行为。

### 7.2 当前缺口：文档与可观测性

- **文档**：项目说明书、core 模块说明、需求对照等文档中**未明确写出**“撮合遵循价格优先、时间优先”，也未标明实现位置（引擎 + 订单簿结构）。
- **可观测性**：前端与接口只展示挂单列表、成交回报与历史订单，**没有**“盘口深度”或“同价订单排队顺序”的展示，因此用户无法直观验证时间优先；若不做专项测试，也难以从行为上区分是否按 FIFO 撮合。

### 7.3 改进方案建议

| 改进项 | 说明 |
|--------|------|
| **1. 文档显式声明** | 在《项目说明书》或《core 模块功能与接口说明》中增加一小节「撮合规则」，写明：本系统撮合由 exchange-core 完成，遵循**价格优先、时间优先**；价格优先体现为买卖盘按最优价档优先撮合，时间优先体现为同一价格档内按挂单先后顺序（FIFO）成交；实现位置为 exchange-core 的 OrderBookNaiveImpl（价格档 TreeMap 顺序 + 同档 LinkedHashMap 顺序）。 |
| **2. 需求追溯** | 在《需求功能与模块对照》中，将「模拟撮合」与「价格优先、时间优先」对应到 **exchange-core / OrderBookNaiveImpl / OrdersBucketNaive**，便于评审与回归。 |
| **3. 单元/集成测试（可选）** | 在 **exchange-core** 或 **trading-core** 的测试中增加用例：<br>• **时间优先**：同一品种、同一价格、同一方向依次下多笔订单，再下一笔反向订单吃尽该档；断言成交回报中 maker 的成交顺序与挂单顺序一致（先挂先成）。<br>• **价格优先**：在多个价格档挂单后，下一笔市价/限价单；断言先成交最优价档、再成交次优价档。 |
| **4. 盘口或排队顺序展示（可选）** | 若需让“时间优先”对用户可见，可增加：<br>• 引擎 L2 快照：通过 **ApiOrderBookRequest** 拉取买卖档深度（含档位量），和/或<br>• 同价订单排队顺序：在业务侧按「价格 + 挂单时间」展示当前挂单列表（需引擎或 Core 提供按价格档与时间排序的挂单查询，或从 L2 推断）。 |

若仅需“在文档中体现”而不改代码，完成 **1 + 2** 即可；若需可验证、可演示，建议再增加 **3**，必要时再考虑 **4**。
