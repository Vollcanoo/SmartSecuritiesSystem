# 功能改进说明

本文档说明对用户管理和交易下单/撤单功能的改进。

---

## 一、用户管理模块改进

### 1.1 创建用户改进

**改进前**：
- 用户需要手动输入用户ID（uid）、股东号（shareholderId）、用户名
- 容易出现格式不统一、重复等问题

**改进后**：
- **用户只需输入用户名**
- **系统自动生成用户ID（uid）**：从数据库查找最大uid，+1递增（起始值10000）
- **系统自动生成股东号（shareholderId）**：格式 `SH` + 8位数字（如 `SH00000001`），与uid对应
- 用户名不能重复，创建时自动校验

**接口变更**：
- **请求**：`POST /api/users`
  ```json
  {
    "username": "张三"
  }
  ```
- **响应**：返回创建的用户信息（含系统生成的uid和shareholderId）
  ```json
  {
    "code": 0,
    "data": {
      "uid": 10001,
      "shareholderId": "SH00000001",
      "username": "张三",
      "status": 1,
      ...
    }
  }
  ```

### 1.2 修改用户名功能

**新增功能**：
- 用户可以修改自己的用户名（但不能重复）
- 用户ID和股东号不允许修改（由系统维护）

**接口**：
- `PUT /api/users/{uid}/username`
  ```json
  {
    "username": "新用户名"
  }
  ```

### 1.3 查询用户余额功能

**新增功能**：
- 支持查询用户余额（资金余额和持仓余额）

**接口**：
- `GET /api/users/{uid}/balance`
- **响应**：
  ```json
  {
    "code": 0,
    "data": {
      "uid": 10001,
      "quoteBalance": 1000000.0,
      "baseBalance": 10000.0,
      "note": "当前为简化实现，所有用户共享引擎余额"
    }
  }
  ```

**说明**：当前实现为简化版，所有用户共享引擎余额（引擎固定uid=1）。实际生产环境需要根据shareholderId映射到不同的引擎uid。

## 二、下单/撤单业务改进

### 2.1 下单改进

**改进前**：
- 用户需要手动输入订单编号（clOrderId）
- 容易出现格式不统一、重复等问题

**改进后**：
- **系统自动生成订单号**：格式 `ORD` + 10位时间戳（秒）+ 4位序号（如 `ORD17370432000001`）
- 用户不需要提供clOrderId（如果提供了也会使用，但建议不提供）
- **下单成功后返回订单编号**，供后续撤单使用

**接口变更**：
- **请求**：`POST /api/order`
  ```json
  {
    "market": "XSHG",
    "securityId": "600030",
    "side": "B",
    "qty": 100,
    "price": 10.5,
    "shareholderId": "SH00000001"
  }
  ```
  **注意**：clOrderId 字段可选，不提供则由系统自动生成

- **响应**：订单确认回报中包含系统生成的clOrderId
  ```json
  {
    "clOrderId": "ORD17370432000001",
    "orderId": 123,
    "market": "XSHG",
    ...
  }
  ```

### 2.2 撤单改进

**改进前**：
- 用户需要提供撤单编号（clOrderId）、原订单编号（origClOrderId）、市场、股票代码、股东号、买卖方向等多个字段
- 容易出错，用户体验差

**改进后**：
- **用户只需提供原订单编号（origClOrderId）**
- **系统自动生成撤单号**：格式 `CAN` + 10位时间戳（秒）+ 4位序号
- **其他字段自动填充**：从挂单记录（OpenOrderStore）中根据origClOrderId自动获取shareholderId、market、securityId、side等

**接口变更**：
- **请求**：`POST /api/cancel`
  ```json
  {
    "origClOrderId": "ORD17370432000001"
  }
  ```
  **注意**：clOrderId、market、securityId、shareholderId、side 字段可选，系统会自动从挂单记录获取

- **响应**：撤单确认回报中包含系统生成的撤单号
  ```json
  {
    "clOrderId": "CAN17370432010002",
    "origClOrderId": "ORD17370432000001",
    "canceledQty": 100,
    "cumQty": 0,
    ...
  }
  ```

## 四、订单号生成规则

### 4.1 订单号格式

- **下单订单号**：`ORD` + 10位时间戳（秒）+ 4位序号
  - 示例：`ORD17370432000001`
  - 长度：18字符
  - 唯一性：时间戳 + 序号保证唯一

- **撤单号**：`CAN` + 10位时间戳（秒）+ 4位序号
  - 示例：`CAN17370432010002`
  - 长度：18字符

### 4.2 股东号格式

- **格式**：`SH` + 8位数字（不足补0）
- **示例**：`SH00000001`、`SH00000002`
- **与用户ID对应**：股东号 = `SH` + 8位uid（如uid=10001 → `SH00010001`）

**注意**：当前实现中，股东号格式为 `SH` + 8位uid，如果uid超过8位，会截断或调整格式。实际实现中，股东号基于uid生成，确保唯一性。

## 五、兼容性说明

### 5.1 向后兼容

- **下单接口**：如果用户提供了clOrderId，系统会使用用户提供的值（但建议不提供，让系统生成）
- **撤单接口**：如果用户提供了其他字段（market、securityId等），系统会校验是否与挂单记录一致；如果未提供，则自动从挂单记录获取

### 5.2 迁移建议

- **现有用户**：可以继续使用，但建议逐步迁移到新接口（不提供clOrderId）
- **新用户**：直接使用新接口，享受自动生成功能

## 七、注意事项

1. **订单号唯一性**：系统生成的订单号基于时间戳和序号，理论上不会重复。如果系统重启，序号会重置，但时间戳不同，仍能保证唯一性。

2. **股东号格式**：股东号格式为 `SH` + 8位数字，与用户ID对应。如果用户ID超过8位，需要调整生成逻辑。

3. **余额查询**：当前为简化实现，所有用户共享引擎余额。实际生产环境需要扩展引擎支持多用户，或维护用户余额映射表。

4. **挂单记录**：撤单时系统从内存中的挂单记录（OpenOrderStore）获取原订单信息。如果订单已完全成交或系统重启，挂单记录可能不存在，此时撤单会失败。

5. **前端适配**：前端页面已更新，适配新的接口。如果使用旧版本前端，需要更新。

## 八、后续优化建议

1. **订单号持久化**：将订单号生成器状态持久化到数据库，避免系统重启后序号重置
2. **多用户余额**：扩展引擎支持多用户，或维护用户余额映射表
3. **订单号格式优化**：考虑使用更短的格式，或支持自定义格式
4. **批量操作**：支持批量下单、批量撤单
5. **订单号查询**：支持根据订单号查询订单详情
