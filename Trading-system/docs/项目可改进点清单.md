# 项目可改进点清单

以下为整体审查后的改进建议；**已按序落实**，并已更新代码与项目文档（项目说明书、测试API示例）。若需进一步调整可说明具体条目。

---

## 已落实说明（简要）

- **一、安全**：Admin `/internal/order-events` 支持 `X-Internal-Token` 鉴权（配置 `trading.admin.internal-token`）；Core 推送时带同 token；DB 密码支持 `DB_PASSWORD` 环境变量；说明书已注明接口无鉴权仅适合内网/测试。
- **二、健壮性**：事件体 null 返回 400；RestTemplate 连接/读取超时 2s/5s 且为 Bean；Gateway 统一错误码 INVALID_JSON/BAD_REQUEST/INTERNAL_ERROR；Admin/Core 增加 `@RestControllerAdvice` 统一 JSON 异常。
- **三、校验**：Core 撤单增加 `CancelValidation`（origClOrderId 非空与长度、股东号与挂单一致）；CANCELLED 时库中无订单打 INFO 日志。
- **四、配置运维**：start-all.sh 中 Admin 端口等待改为 90 秒；说明书与测试 API 文档补充多机部署、健康检查、ddl-auto 生产建议。
- **五、可观测**：Core 下单/撤单日志带 requestId；Admin 事件处理成功打 debug 日志；Admin/Core/Gateway 暴露 `/actuator/health`。
- **六、代码**：DatabaseConfig 去除重复 `@EnableJpaRepositories`；OrderEventSender 使用 `adminRestTemplate` Bean；Gateway 注入 ObjectMapper；OrderHistoryService 使用 EVENT_* 常量。
- **七、文档**：说明书补充订单历史在 Admin、数据流、多机与健康检查；测试 API 补充 internal 鉴权说明、健康检查、历史分页参数。
- **八、功能**：订单历史 API 支持 `page`、`size` 分页（默认 0、50，最大 500）；8.1 成交回报驱动状态、8.3 股东号校验为可选扩展，见下方原表。

## 一、安全与访问控制

| 序号 | 问题 | 建议 | 优先级 |
|------|------|------|--------|
| 1.1 | **Admin 的 `/internal/order-events` 无鉴权**：任何能访问 8080 的请求都可伪造订单事件写入 DB。 | 增加简单鉴权：如请求头 `X-Internal-Token` 与配置中的密钥一致才处理；或仅允许来自本机/内网 IP。 | 高（若 Admin 暴露在外网） |
| 1.2 | **Admin/Core 的 REST 接口均无登录/鉴权**：用户、订单历史、下单、撤单等谁都能调。 | 若需多租户或生产环境，可增加 API Key、JWT 或统一网关鉴权；当前 demo 可保持现状并文档说明「仅内网/测试」。 | 中 |
| 1.3 | **MySQL 密码写在 application.yaml** | 生产建议用环境变量或配置中心，如 `spring.datasource.password: ${DB_PASSWORD}`。 | 低（当前多为空密码本地用） |

---

## 二、健壮性与异常处理

| 序号 | 问题 | 建议 | 优先级 |
|------|------|------|--------|
| 2.1 | **InternalOrderEventController**：`event` 为 null 时 `event.getClOrderId()` 会 NPE。 | 方法开头判空，`if (event == null) return 400` 或统一返回 400 body。 | 高 |
| 2.2 | **OrderEventSender**：`RestTemplate` 未设置连接/读取超时，Admin 慢或挂掉时 Core 可能长时间阻塞。 | 给 RestTemplate 配置 connectTimeout/readTimeout（如 2s/5s），或使用 `RestTemplateBuilder` 注入。 | 中 |
| 2.3 | **Gateway**：`channelRead0` 里 JSON 解析失败会抛异常，虽在 `exceptionCaught` 里处理，但异常信息直接写回客户端可能暴露内部细节。 | 对解析/业务异常返回统一格式（如 `{"error":"INVALID_JSON"}`），避免把堆栈或内部消息写给客户端。 | 中 |
| 2.4 | **Core/Admin 无全局异常处理**：Controller 未捕获的异常会变成 Spring 默认 500 页。 | 增加 `@RestControllerAdvice` + `@ExceptionHandler`，统一返回 JSON 错误体（code、message），便于前端/调用方处理。 | 中 |

## 三、校验与一致性

| 序号 | 问题 | 建议 | 优先级 |
|------|------|------|--------|
| 3.1 | **撤单请求未做业务校验**：Core 只根据 `origClOrderId` 查引擎 ID，未校验 `market/securityId/shareholderId/side` 是否与原单一致。 | 在 Core 增加 CancelRequest 基本校验：origClOrderId 非空、长度等；若有 OpenOrderStore 可校验股东号与方向是否与挂单一致，防止误撤。 | 中 |
| 3.2 | **下单前未校验股东号是否在 Admin 存在且启用**：当前仅对敲风控用股东号，引擎侧共用 uid。 | 若希望「仅 Admin 已创建且启用的股东号可下单」，Core 可调 Admin 查询用户接口或本地缓存一份「有效股东号」并校验。 | 低（当前设计是引擎不区分股东号） |
| 3.3 | **OrderHistoryService.processEvent**：CANCELLED 时若 `findByClOrderId` 不存在则静默忽略，无日志。 | 打一条 debug/INFO 日志（如「收到 CANCELLED 但库中无此 clOrderId」），便于排查 Core/Admin 事件顺序或丢事件问题。 | 低 |

## 四、配置与运维

| 序号 | 问题 | 建议 | 优先级 |
|------|------|------|--------|
| 4.1 | **Core 的 `trading.admin.base-url` 写死 localhost**：多机部署时 Admin 未必在本机。 | 保持默认 `http://localhost:8080`，在文档或启动说明中写明「多机时需改配置或环境变量」。 | 低 |
| 4.2 | **Gateway 的 Core 地址**：同样写死 localhost，多机需改配置。 | 同上，文档说明即可；若有需要可加「启动前检查 Core 是否可达」的简单健康检查。 | 低 |
| 4.3 | **JPA ddl-auto: update**：生产环境自动改表有数据风险。 | 生产建议改为 `validate` 或 `none`，表结构用 Flyway/Liquibase 管理；当前 demo 可保留 update。 | 中（上生产时） |
| 4.4 | **start-all.sh 中 Admin 等待端口 20 秒**：若 MySQL 慢或首次建表多，20 秒可能不够。 | 将 Admin 的 `wait_for_port` 超时调大（如 90），或与 Core 一致，减少「误判启动失败」。 | 低 |

---

## 五、可观测与排障

| 序号 | 问题 | 建议 | 优先级 |
|------|------|------|--------|
| 5.1 | **无统一请求日志**：难以追踪「某笔订单从 Gateway → Core → 引擎」的完整路径。 | 在 Gateway/Core 为每笔请求生成或传递 `requestId`（如 UUID），打 INFO 日志（requestId、clOrderId、结果），便于 grep 排障。 | 中 |
| 5.2 | **Admin 订单事件接收无成功日志**：只有失败时 warn。 | 收到事件并 processEvent 成功后打一条 debug 日志（clOrderId、eventType），便于确认 Core→Admin 推送正常。 | 低 |
| 5.3 | **无健康检查接口**：运维/网关无法判断服务是否真正就绪（如 Core 是否已连引擎）。 | Admin/Core/Gateway 暴露 Spring Boot Actuator `health`（或自建 `/health`），返回 200 表示就绪；Core 的 health 可包含「引擎是否已初始化」。 | 中 |

## 六、代码与结构

| 序号 | 问题 | 建议 | 优先级 |
|------|------|------|--------|
| 6.1 | **DatabaseConfig 与 TradingAdminApplication 重复**：`@EnableJpaRepositories` 在两边都出现。 | 保留在主类上的 `@EnableJpaRepositories`，从 DatabaseConfig 中移除，避免重复。 | 低 |
| 6.2 | **OrderEventSender 中 new RestTemplate()**：未交给 Spring 管理，不利于后续统一配置（超时、拦截器）。 | 改为 `@Bean` 或在配置类中构造 `RestTemplate` 并注入，便于统一设置超时等。 | 中 |
| 6.3 | **Gateway ObjectMapper 每次 new**：可复用单例。 | 使用 `ObjectMapper` 的 `@Bean` 注入，或使用 Spring 默认的 ObjectMapper。 | 低 |
| 6.4 | **OrderHistoryService 中魔法字符串**：`"PLACED"`、`"CANCELLED"` 等散落。 | 与常量 `STATUS_LIVE` 等一致，用常量或枚举表示 eventType，减少拼写错误。 | 低 |

## 七、文档与使用

| 序号 | 问题 | 建议 | 优先级 |
|------|------|------|--------|
| 7.1 | **项目说明书未体现「订单历史在 Admin」** | 在「数据流」或「模块职责」中补充：订单事件由 Core 推 Admin，历史订单表在 Admin，查询走 Admin 的 `/api/orders/history`。 | 低 |
| 7.2 | **测试 API 文档未写 Internal 接口** | 明确说明 `/internal/order-events` 仅给 Core 调用，不建议手工调用；若做了 1.1 鉴权，可写「需带 X-Internal-Token」。 | 低 |

## 八、功能扩展（可选）

| 序号 | 方向 | 说明 | 优先级 |
|------|------|------|--------|
| 8.1 | **成交回报驱动订单状态** | 当前仅在下单回报、撤单时更新状态；若引擎有后续成交（部分成交、全部成交），Core 可消费 resultQueue 向 Admin 发 PARTIALLY_FILLED/FILLED，使历史订单的 filledQty/status 更准确。 | 中（若需精确持仓与历史） |
| 8.2 | **Admin 分页与排序** | `/api/orders/history` 当前仅 limit，可增加 `page`、`sort=createdAt,desc` 等，便于前端分页。 | 低 |
| 8.3 | **Core 按股东号校验** | 若将来引擎支持多 uid，Core 可根据 shareholderId 映射到不同 uid，并在下单前校验该股东号在 Admin 已存在且启用。 | 低 |
